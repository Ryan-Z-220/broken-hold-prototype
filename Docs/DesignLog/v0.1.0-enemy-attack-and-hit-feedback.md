# v0.1.0 — Enemy Attack Detection and Player Hit Feedback

## Feature Name

Enemy Spherical Attack Detection and Player Hit Feedback

## Design Goal

The purpose of this feature is to turn enemy attacks from simple distance-based behavior into spatially verifiable actions.

Entering the `Attack` state does not automatically guarantee a hit. Whether an attack succeeds should depend on whether the attack volume overlaps the player's collision body at the moment the attack is performed.

This creates a clearer separation between:

* deciding to attack;
* performing the attack;
* determining whether the attack hits;
* applying hit feedback.

## Problem to Solve

In the earlier prototype, enemies only checked whether the player was within a certain distance and then printed an attack message.

This caused several design problems:

1. Enemy attacks had no real spatial hit detection.
2. Entering attack range effectively guaranteed a successful attack.
3. The player could not avoid an attack by moving outside its active range.
4. Attack initiation distance and actual hit range were treated as the same value.
5. It was difficult to visually explain why an attack hit or missed.
6. The system could not easily support different weapon lengths or attack shapes.

## Player Experience Benefits

### Clearer and More Understandable Combat

The player can understand why an attack hits by observing the actual attack range instead of experiencing an invisible distance-based hit.

### Opportunity to Avoid Attacks

If the player moves outside the active attack volume before the attack check occurs, the attack can miss.

This creates the foundation for future mechanics such as:

* positioning;
* dodging;
* sprinting away;
* attack anticipation;
* invincibility frames.

### Improved Combat Fairness

Attack intention and attack result are now separate steps.

An enemy can decide to attack, but the attack only succeeds when the player's collision body is inside the active attack volume.

This makes combat rules easier for the player to learn and predict.

### Better Debug Visibility

The attack sphere can be displayed with Gizmos, allowing the developer to compare the visible attack range with the actual collision result.

## Current Rules

1. The enemy enters the `Attack` state when the player is within `attackDistance`.
2. When the attack cooldown reaches zero, the enemy performs one attack.
3. The attack uses `attackPoint` as the center of the collision check.
4. The attack uses `attackRadius` as the spherical hit range.
5. The collision check is performed once with `Physics.OverlapSphere`.
6. If the sphere overlaps the player's collider, the attack is considered successful.
7. If no player collider is detected, the attack is considered a miss.
8. The player currently has no health system.
9. A successful hit produces:

   * a Console message;
   * a temporary color flash on the player.
10. One attack can only trigger one player hit, even if the player has multiple colliders.

## Technical Implementation

The enemy contains an `AttackPoint` child object that represents the center of the attack volume.

The same attack center is used by both:

* the actual `Physics.OverlapSphere` hit detection;
* the Gizmo visualization.

This ensures that the visual debugging representation matches the real gameplay logic.

The attack process follows this order:

```text
Enemy enters Attack state
→ Attack cooldown expires
→ Attack collision check is performed
→ Hit or miss result is determined
→ Result is logged
→ Player receives hit feedback
```

A helper method is used to ensure that the attack logic and Gizmo visualization always reference the same position:

```csharp
private Vector3 GetAttackCenter()
{
    return attackPoint != null
        ? attackPoint.position
        : transform.position + transform.forward * 1.0f;
}
```

If `attackPoint` is not assigned in the Inspector, the system uses a fallback position in front of the enemy.

## Current Version Scope

The v0.1.0 version intentionally uses a simple spherical collision check.

The following systems are not included yet:

* attack animations;
* animation-timed active frames;
* persistent hitboxes;
* weapon-shaped hitboxes;
* player health;
* player death;
* blocking;
* parrying;
* dodging;
* invincibility frames;
* attack interruption;
* wall obstruction checks;
* multiple attack types.

The goal of this version is to validate the basic relationship between:

```text
Attack State
→ Attack Volume
→ Collision Result
→ Player Feedback
```

## Known Limitations

1. The attack can currently pass through walls.
2. The hit volume is spherical rather than matching a weapon trajectory.
3. The attack has no startup, active, or recovery phase.
4. The player cannot currently lose health or die.
5. Player collider size affects whether the attack is detected.
6. `AttackPoint` must be assigned manually in the Unity Inspector.
7. The system does not currently distinguish between body parts or different hurtboxes.
8. The enemy does not currently rotate toward the player during an attack animation because no animation system exists yet.

## Future Extensions

Planned improvements include:

* converting attacks into reusable `AttackEvent` objects;
* recording the attacker, intended target, actual target, damage, and hit result;
* adding attack startup, active, and recovery timing;
* separating hitboxes from hurtboxes;
* checking whether walls block the attack;
* adding player health and death;
* adding dodge invincibility frames;
* adding blocking and parrying;
* supporting different attack shapes;
* supporting weapon-specific ranges;
* supporting attacks between players, enemies, and NPC factions;
* allowing witnesses to react to attack events;
* using the same attack system for faction conflicts and friendly fire.

## Design Summary

This feature changes enemy attacks from automatic distance-based damage into spatially testable actions.

The player is now hit because their collision body overlaps the enemy's active attack volume, not simply because the enemy is nearby.

This provides a stronger foundation for readable, fair, and expandable action combat.
